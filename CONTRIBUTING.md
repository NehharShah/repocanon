# Contributing to RepoCanon

Thanks for your interest in RepoCanon. This project aims for a small, sharp v1: deterministic analysis, idiomatic outputs per AI tool, and no required network calls.

## Local setup

```bash
git clone https://github.com/NehharShah/repocanon
cd repocanon
python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -e ".[dev]"
```

## Running checks

```bash
make test        # pytest
make lint        # ruff check
make typecheck   # mypy
make format      # ruff format + ruff --fix
```

Every PR should keep `make test` and `make lint` green.

## Adding a framework detector

1. Add a `_Rule` to the appropriate list in `repocanon/analyzer/framework_detect.py`.
2. Add a fixture file (or extend an existing one) under `tests/fixtures/` so the detector has a concrete repo to bite on.
3. Add an assertion in `tests/test_analyzer.py`.

Detectors should err on the side of missing a framework rather than inventing one. If you're not sure, mark the finding as `Confidence.medium` or `low`.

## Adding a new generator target

1. Create a module in `repocanon/generators/<target>.py` that exports a `generate_<target>(model, existing="") -> GenerationPlan` function.
2. Wire it into `repocanon/generators/__init__.py` and `repocanon/cli.py`.
3. Add a snapshot-style assertion in `tests/test_generators.py` that checks for repo-grounded content (specific strings from the model).
4. Document the new target in the README.

Generators must:

- only read from the `ProjectModel`
- omit empty sections rather than padding with filler
- include the `<!-- Generated by RepoCanon -->` header
- include a `<!-- repocanon:manual:* -->` block for single-file targets so user edits survive regeneration

## Style

- Keep public functions documented in plain English.
- Comments explain non-obvious intent, not what the code already says.
- Prefer adding tests to lengthening prose.

## Fixture repos

The tests rely on tiny, realistic fixture repos under `tests/fixtures/`. Keep them small but representative ‚Äî they double as visible documentation of what RepoCanon can detect.

## Releasing

Releases are automated. Pushing a tag matching `v*` triggers `.github/workflows/release.yml`, which:

1. Runs `make lint`, `make typecheck`, `make test` on Python 3.11 / 3.12 / 3.13.
2. Verifies the git tag matches the `version` field in `pyproject.toml`.
3. Builds the sdist + wheel with `python -m build` and validates them with `twine check`.
4. Uploads to PyPI via [Trusted Publishing](https://docs.pypi.org/trusted-publishers/) ‚Äî no API tokens are stored in the repo. The `pypi` GitHub Environment requires manual approval before the upload step runs.

To cut a release:

```bash
# bump version in pyproject.toml
git commit -am "Release vX.Y.Z"
git tag -a vX.Y.Z -m "RepoCanon X.Y.Z"
git push origin main vX.Y.Z
# then approve the publish job on github.com/NehharShah/repocanon/actions
```
